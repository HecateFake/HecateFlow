# 自主性优先多模型编排契约 / Autonomy-First Multi-Model Contract

HecateFlow skill 的职责是主动提升模型能力,而不是把不确定性转嫁给用户。默认流程是:先自主求证,自动读码、搜索、查 manifest、看 git diff、运行只读命令;再按复杂度主动派发只读子代理扩大证据覆盖;高风险结论走复审链;最后由主 agent 亲验并裁决。只有外部不可证事实、安全边界、硬件/物理风险、写入 worker 越权风险、Git/远端历史变更才需要用户确认。

多模型协作的目标不是增加写手数量,而是把不同模型放进不同权限层级:主 agent 持权,持有上下文、代码主权、用户沟通、最终裁决和 Git 权限;只读子代理负责探索、规划、审查、复核,由主 agent 自动主动派发,不需要用户额外确认;写入型 worker 只在用户已明确要求实现/修改/落地/应用补丁、方案完整、范围互斥且边界清楚后短时执行。所有 HecateFlow skill 默认继承本契约。

## 自主性分级

| 等级 | 含义 | 行为 |
|------|------|------|
| A0 自主发现 | 代码、manifest、docs、git diff、命令输出可发现的事实 | 主 agent 必须先读、搜、跑只读命令,不得把可发现事实问给用户 |
| A1 自主增智 | L1-L3 需要额外视角或高风险证据 | 主 agent 主动派发只读子代理和复审子代理;不需要额外用户确认 |
| A2 自主实施 | 用户已明确要求实现/修改/落地/应用补丁 | 主 agent 可进入写入主流程;实施 worker 仅在方案确定、文件范围互斥、边界清楚时使用 |
| A3 必须确认 | 外部不可证事实、安全边界、硬件/物理风险、写入 worker 越权风险、Git/远端历史变更 | 最小提问,只问无法自主证明且会改变风险边界的事项 |

## 角色与权限

| 角色 | 能做 | 禁止 |
|------|------|------|
| 主 agent | 读 manifest/文档/代码,整合证据,裁决方案,执行或整合写入,运行验证,向用户汇报 Git 建议 | 跳过证据亲验,自动提交/推送,回退非本次改动 |
| 只读子代理 | 搜索、读码、对比、分析、规划、审查、评估、提出建议;由主 agent 按复杂度主动派发 | 写文件、应用 patch、格式化、stage、commit、push、扩大任务 |
| 复审子代理 | 检查其它子代理结论的证据充分性、矛盾、过度推断和遗漏红线 | 直接替主 agent 裁决或修改文件 |
| 实施 worker | 在指定文件范围内执行已定方案,返回 diff/风险/验证结果 | 跨范围、改规则边界、提交/推送、把探索问题变成实施问题 |

## 复杂度矩阵

| 等级 | 场景 | 编排要求 |
|------|------|----------|
| L0 简单局部 | 单文件小改、低风险文档措辞、证据直接可见 | 主 agent 自查即可;仍保留 Git 确认门 |
| L1 中等 | 多文件文档/模板同步、局部代码改动、需要找引用或对比前后语义 | 自动主动派发至少 1 个只读 reviewer 查遗漏或风险;主 agent 亲验关键证据 |
| L2 高风险 | 跨 target、ISR/volatile、协议、共享内存、安全、删除、行为保持重构、规则治理 | 自动多路只读分工(探索/架构/审查),复审子代理查证据链,主 agent 亲验后裁决 |
| L3 系统级 | 多阶段跨 target、构建图/LSP/文档矩阵同时变化、难复现底层问题、全包规则重构 | 自动只读调研和计划,复审链闭合后形成决策完整方案;需要时才分派互斥范围 worker |

## 最小提问原则

- 先自主求证:能从仓库、manifest、文档、日志、diff、命令输出得到的事实,不得先问用户。
- 问题只用于改变风险边界:外部不可证事实、物理接线/极性/上板现象、安全边界、写入 worker 越权风险、Git/远端历史变更。
- 提问前必须说明已查证据和剩余不确定性;不要把"我还没查"包装成用户问题。
- 平台没有子代理工具时,主 agent 降级为本地只读命令亲验,并说明"平台限制未做独立子代理复核"。

## 子代理输出契约

只读子代理报告必须包含:

- **范围**:检查了哪些文件、模块、问题或命令。
- **证据**:路径、符号、行号、命令或 diff 依据。
- **结论**:PASS/FAIL/风险/建议,避免泛泛判断。
- **置信度**:高/中/低,说明不确定来源。
- **未决点**:需要主 agent 或用户裁决的事项。

复审链闭合时,复审子代理额外检查:

- 原子结论是否有证据支撑。
- 多个子代理是否互相矛盾。
- 是否把推测、历史规则、用户描述或 SDK 承诺当当前事实。
- 是否遗漏 Git 确认门、文档同步、场景约束、跨 target 边界或安全红线。

## 子代理并发槽位纪律

- 主 agent 派发前先检查现有子代理状态,关闭已完成或不再需要的代理,防止占满并发上限。
- L2/L3 多路只读按批次派发;保留至少一个可用槽位给复审子代理或紧急核验,不要一次性把全部并发槽位占满。
- `wait` 到子代理结论后,先吸收范围/证据/结论/未决点,再及时 `close` 释放槽位;不得让已完成子代理长期占用并发额度。
- 若平台并发上限已满,主 agent 应先关闭已完成代理或分批等待,再继续派发;不能把并发上限写成把只读派发转嫁给用户确认的理由。
- 最终汇报若因平台并发限制改为分批或顺序复审,须说明这是槽位/平台限制,不是 HecateFlow 策略降低复审强度。

## 实施 worker 门

只有全部满足时才可委派写入型 worker:

- 用户已明确要求实现/修改/落地/应用补丁,任务不是开放式探索。
- 主 agent 已吸收只读调研,形成决策完整方案。
- 每个 worker 的文件所有权互斥,目标文件、改动边界、接口/行为预期、验证方式和禁止事项写清。
- worker 被明确告知不得提交、推送、跨范围、修改规则边界或扩展任务。
- 若 worker 范围不互斥、接口预期不清、可能越过规则/安全边界,必须先向用户确认或改由主 agent 本地实施。

worker 返回后,主 agent 必须读 diff、运行必要检查,并对安全/行为/删除/协议/规则类关键点亲自复核。

## Git 确认门

- 子代理和 worker 永不拥有 Git 写权限。
- 主 agent 完成实现和验证后只输出变更摘要、验证结果、建议提交说明和待暂存文件清单。
- 只有用户明确确认提交/推送这一组改动后,才可显式 `git add <file...>`、`git commit`,并按 manifest `git.remotes` 推送。
- 禁止 `git add .` / `git add -A`;禁止自动提交或自动推送。

## 平台降级

- Claude Code:可用 `Task` 并行只读探索/审查/复审;只读派发主动进行,写入 worker 仍受本契约约束。
- Codex:多代理工具可用时主动使用 `multi_agent_v1.spawn_agent` / `wait_agent` / `close_agent` 派只读 explorer/reviewer/reviewer-of-review;若宿主工具策略限制无显式请求派代理,主会话顺序自审,并在结果中声明平台限制导致未做独立子代理复核。
- Reasonix/Qoder/其它 CLI:按实际暴露工具执行;无等价工具时回退为主 agent 自审,不得伪称已多模型复核。

## 验收场景

- 行为保持重构:只读对抗审查 + 复审 + 主 agent 亲验。
- 跨核/ISR/协议/安全任务:L2/L3 多路只读调研 + 复审子代理 + 主 agent 亲验,写入 worker 后置。
- 简单 L0 改动:主 agent 自查可接受,但仍执行 auto-workflow 与 Git 确认门。
- 过度提问审查:若事实可由仓库/manifest/命令发现却先问用户,视为自主性缺陷,应记录 lesson 或升级规则。
