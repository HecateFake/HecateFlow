# hf-hw-mapping — 硬件映射 / 极性 / 数量级 / Hardware Mapping, Polarity & Magnitude

嵌入式工程里**最贵的 bug 不在算法,而在"引脚散落、极性藏错地方、增益量纲混用"**。这三件事一旦在源码里随处硬编码,换一次接线、改一次车体就要全工程翻找,还容易在大电流执行器上酿成物理事故。本 skill 把它们固化为三件可复用的工程纪律:**头文件分层组织、极性单一真相源、数量级理智检查**,并定义 agent 在触及这些点时的**主动提醒-确认**义务。被 `hf-init-project`(脚手架)、`hf-design-module`(放置/检查)、`hf-auto-workflow`/`hf-embedded-safety`(审查)引用,也可单独调用("帮我把引脚集中""这个极性该放哪")。

## 适用 / 不适用

- 适用:组织引脚/参数头、新增驱动接线、辨识或修改执行器/编码器/电流极性、整定 PID 增益与菜单步长、设计涉独占 IO 外设的模块。
- 不适用:纯算法正确性(普通 review)、纯并发/钳位/失控安全(去 `hf-embedded-safety`)、纯编码风格(见 `../references/embedded-c-style.md`)。

## 触发关键词

引脚映射 / pinMap / 硬件映射头 / 参数头 / configHeader / 非扁平布局 / 极性 / 方向系数 / polarity / 正反馈 / 数量级 / 增益 / 步长 / magnitude / IO 外设归属 / hardware map。

## 第一性原则

**"硬件契约"(引脚、极性、量纲)要从算法里抽出来,集中到可辨识、可改、爆炸半径有限的单一位置。** 三条推论:

- **引脚是 HAL 事实,参数是整定事实,算法是逻辑——三层分离。** 引脚走 `pinMap.h`、可调参数走 `configHeader`、算法只依赖抽象接口。换芯片/换接线/换车体时**只动底层一处,算法层不改**(对齐 `../references/embedded-c-style.md` 的"抽象分层与可移植封装")。
- **极性属"参数"不属"引脚",且必须有唯一归属。** 物理方向(转向/反馈符号)是接线、安装、旋向与传感器决定的事实,集中在 `configHeader` 的 **§极性段**(每路执行器/传感器通道各一个可调方向系数宏,个数随硬件),**绝不分散藏进各驱动 .c 或 PID Kp 符号**。
- **量纲与数量级是隐形契约。** 不同控制环、不同物理量的增益差几个数量级(速度环 Kp~23 vs 电流环 Kp~0.002,差 ~1e4),菜单步长必须与默认值/范围/钳位同量级。混用即失控或无响应。

## 红线(最易翻车)

- **极性藏进 PID Kp 负号**:把执行器/反馈翻转藏在 Kp 符号里,Kp 身兼**增益 + 极性**二职——反直觉,且误改一个符号即把负反馈变正反馈,大电流执行器瞬间跑飞伤人/烧驱动。**极性只许在 §极性段的方向系数宏(每路通道一个,个数随硬件),Kp 全用正号。**(头号杀手,与 `hf-embedded-safety` 同列。)
- **引脚硬编码散落各 .c**:同一引脚在多个驱动里各写一遍,改接线漏改一处→ 行为诡异。引脚必须在 `pinMap.h` 唯一定义。
- **方向系数当"固定常量"照抄**:`*_DIR` 的 ±1 是**针对本台硬件实测辨识**的结果,不是可移植常量;换车/重接线照抄旧值 = 极性全错。
- **闭环轴向只靠开环辨识**:开环单执行器辨识只覆盖"正命令→正反馈",**不覆盖**闭环控制器要求的"传感器正向 = 被控量正向"(如 yaw 环要 IMU yawRate 正向 = 车体 CCW 左转)。轴向对齐须**另行手动确认**。
- **数量级失配**:菜单调参步长与参数范围差几个数量级(历史 4 处),按一下要么纹丝不动要么直接打满钳位。

## 一、头文件分层组织(pinMap / configHeader / 算法层三分)

### 非扁平布局放置(消费 manifest `targets[].layout`)

`project/code/` 按功能分子目录(`style:"feature-subdirs"`,`subdirs` 如 `app/control/sensor/comm/config/util`),**不是**一锅端的扁平目录。两类集中头放 **`config/` 子目录**:

```
project/code/
├── config/
│   ├── pinMap.h          # HAL 引脚层:零依赖纯 #define,引脚单一真相源
│   └── configXxx.h       # 参数层:分节编译期宏 + §极性段
├── control/  sensor/  comm/   # 算法/驱动层:从 config/ 取引脚与参数,不硬编码
└── ...
```

新增 `config/` 等子目录须同步构建系统 include 路径与 LSP `-I`(见 `hf-build-sync`、`../references/embedded-c-style.md` 路径纪律)。

### pinMap.h —— HAL 引脚层(引 `../hecateflow/templates/pinMap.h.tmpl`)

- **零依赖纯 `#define`**:不 `#include` 任何驱动头;引脚值是驱动头提供的 token(`TCPWM_CHxx_Pyy` / `ADCz_CHxx` / GPIO 枚举),消费 `.c` 各自 include 对应驱动头展开。这让 pinMap.h **可被任意模块安全 include 而不引入耦合**(最小侵入)。
- 按功能 `/* ===== 块 ===== */` 分组:执行器 PWM+方向、反馈传感器 INDEX+AB 相、模拟采样 ADC、人机接口(按键/蜂鸣/灯)、通信收发器(方向/RX/TX)。
- 命名 `{功能}_{通道/轮/实例}_{类型}`,如 `MOTOR_FL_EN_PWM` / `ENCODER_FR_CH1` / `CURRENT_SENSE_BL_ADC`。
- 易踩的语义就地注释:按键丝印顺序可能与其它板**反向**、收发器方向引脚的**高低电平语义**(高=发送/低=接收)。
- 与跨核引脚冲突矩阵 `docs/PINOUT` 配对:**先改 pinMap.h 再更新矩阵**(pinMap 是源,矩阵是视图)。

### configHeader —— 参数整定层(引 `../hecateflow/templates/config-header.h.tmpl`)

编译期宏单一真相源,**带分节索引注释 + 量纲约定**,分节:
- §A 调度时序与模式(ISR 周期 / 分频 / 构建变体宏)。
- §B/§C/§D 各控制环 PID(多实例则**各实例独立增益**,如四轮各一组 Kp/Ki/Kd)。
- §E 几何 + 限幅 + 滤波。
- **§极性** 三组方向系数(见下"二")。
- §F 通信映射 / 故障保护。

> 源工程实例(只读出处:`core2_cyt4bb7/project/code/config/configMotor.h`)即此结构:§A..§F + §极性,头注开篇写明量纲约定("编码器脉冲/2ms,PID 吸收尺度,无物理换算")。

### 三层为何分离(对齐 embedded-c-style.md §抽象分层 / §嵌入式 OOP)

算法/控制层依赖**抽象接口**,不直接摸引脚/寄存器:引脚走 pinMap、参数走 configHeader、外设访问走薄 HAL 封装。通用能力(PID/滤波/估计)封成**对象式** `xxxStruct` + `Init`/`Update`/`Reset`,多实例隔离状态(四轮各持一个 PID 实例,互不串),换硬件用**函数指针 / init 绑定**注入具体驱动(=多硬件兼容 API)。**接口契约——单位、量纲、极性约定、返回语义——写在头注释里**,作为跨实现的合同。这样用户能低成本换实现/A-B 对比/调参,爆炸半径局限在底层一处。极性正是这套契约的一部分,所以它属"参数层契约"而非散落算法层。

## 二、极性单一真相源(代码↔现实映射宏 + 标定 + 闭环核查 + 轴向对齐)

### 方向系数 = 每路通道的映射宏

**每一路执行器命令、每一路传感器反馈各有一个方向系数宏**,吸收该通道"代码符号↔现实方向"的取号。系数个数 = 通道数,**不是固定三组**;下表为源工程(轮式底盘)的三类实例,换硬件按实际通道增减(多旋翼会有电调旋向系数、姿态环会有 IMU 轴映射符号等):

| 系数(实例) | 乘在哪 | 作用 |
|------|--------|------|
| 执行器输出 `*_OUTPUT_DIR`(浮点 ±1.0f) | driver 乘命令(如 `motorSetSpeed` 乘 speed) | 使 **+命令 → 物理正方向** |
| 编码器 `ENCODER_*_DIR`(整型 ±1) | 乘原始计数 | 使 **正方向 → +速度反馈** |
| 电流 `CURRENT_SENSE_*_DIR`(整型 ±1) | 乘 (raw−offset) | 使 **正方向 → +电流反馈** |
| (姿态环示例)IMU 轴映射符号 | 乘陀螺仪某轴原始值(如 `-gyro_y`) | 使 **被控量正向 → +角速度反馈** |

### 核心:代码符号 ⇄ 现实方向的映射(可调宏)

极性的本质 = **确认每个传感器/执行器"代码里的符号"⇄"现实物理行为方向"的映射**。每个量在代码里是带符号的(读到的计数/角速度/电流,或写出的占空比/油门/转矩),它在现实里对应一个物理方向:

- 执行器:写**正命令**,物体往哪个物理方向动?
- 传感器:物体往某物理方向动,读到的是**正还是负**?(编码器计数、IMU/陀螺仪某轴、电流采样、霍尔…)

这个映射**由硬件决定、因台而异**(接线相序、传感器安装朝向、电调旋向 CW/CCW、轮系排布、舵机行程方向、ADC 采样取反…),因此三条硬要求:

1. **凡可能因硬件变化的映射,必须做成头文件里一个可调的方向系数宏**(`*_DIR` ±1 / 轴映射符号),不写死、不散落各 `.c`;换硬件/重接线只改这一个宏。
2. **agent 不得自行假定映射方向,必须提示用户标定**(开环实测:给正命令看物理方向 / 看反馈符号),把标定结果填进该头宏。
3. 各映射取号自洽后归一化到"选定物理正方向=正"——正命令→正向运动、正向运动→正读数。这是**结果**,它使上层所有控制环 **Kp 全用正号**(闭环参数全为正数),Kp 只承载增益、不承载极性。

> 归一化锚点(不变量):为每个受控自由度选定物理正方向后,该自由度每一路执行器命令、每一路传感器反馈符号都与之一致。**举例(非穷举,按实际硬件归纳勿照抄)**:轮式单轮(占空比/编码器/电流);姿态环(舵量/差速 + IMU 陀螺仪轴,受安装朝向+轴映射影响,如 `-gyro_y`→yawRate);多旋翼(油门 + 电调旋向 CW/CCW + 桨叶正反);全向底盘(各轮方向 + 轮系安装排布 → 整车 vx/vy/wz)。

未标定即整定闭环 = 在未知映射上调正号 Kp,可能正反馈跑飞。

### 开环实测辨识(±1 怎么定)

各 DIR 的 ±1 **不是固定常量**,由**开环实测**逐实例辨识:给正命令(正占空比)→ 观察执行器实际转向、各路反馈符号 → 取使不变量成立的 ±1。**换硬件 / 重接线 / 换车体须重新辨识**。源工程历史:`core2 BL/BR 编码器 INDEX↔引脚` 曾因重新接线而极性翻转,照抄旧值即失控(见反面教训)。

### 闭环轴向对齐(开环辨识不覆盖)

闭环控制器(如 yaw/航向环)**另需**"传感器正向 = 被控量正向":本工程 yaw 环要 IMU `yawRate` 正向 = 麦轮 `wz` 正向(CCW/左转),符号在 `IMU.c`(`-gyro_y`)。开环单执行器辨识**只管单轮命令-反馈**,不管整车旋转的轴向,**必须手动转动车体确认**(左转时 yawRate>0 才对)。

### 闭环自洽判据(改 DIR/Kp 前自检)

同实例各路通道(输出/编码器/电流/IMU 轴…)方向映射自洽,且与 Kp 同号成对成立负反馈:**Kp × 反馈斜率 < 0**。翻任一方向系数等同翻该环极性,须与对应环 Kp 同号成对调整——**改任一系数前必核实物理接线/安装/旋向**。

## 三、数量级理智检查

- **不同环量纲不混用**:速度环 Kp~23(脉冲/2ms 域)与电流环 Kp~0.002(ADC 原始量域)差 ~1e4,各有量纲,**禁止互相照搬**。新增/整定增益前先确认它作用在哪个量纲。
- **菜单步长 vs 范围/默认/钳位**:运行时可调参数的**编辑步长必须与默认值/范围/钳位同量级**。给 Kp=0.002 配步长 1.0 → 一下打满;给 Kp=23 配步长 0.001 → 永远调不动。源工程曾有 4 处步长数量级失配。
- **方向**:增益过大 → 失控/震颤;过小 → 无响应。整定从小到大、看阶跃响应(上升/超调/稳态)。
- **编译期防呆**:量纲/范围约束能用 `#if ... #error` 在编译期挡住的就挡(如摇杆映射分母 `(RANGE−DEADZONE)>0` 防除零),比运行时才炸好。

## 四、主动提醒-确认义务(对应 `activeChecks.polarityMagnitude`)

agent **触及以下任一**时,**不得静默改动、不得自行假定极性**,必须在回复里**显式向用户提问、请其确认物理事实**:

- **标定前置——主动要求用户先完成"代码↔现实"映射标定**:整定任何闭环前,先**提问让用户确认/完成**该自由度上每个执行器与传感器的方向映射标定——对每个量确认"代码里的正值对应现实哪个物理方向"(执行器:正命令往哪动;传感器:某物理方向读到正还是负)。映射若因硬件可变,**确保它落在头里一个可调方向系数宏**(`*_DIR` ±1 / 轴映射符号),用户标定值填入此宏,不写死、不散落 `.c`。**映射未标定,不在未知映射上整定闭环。**
- 改/读 §极性段任一 `*_DIR`,或新增执行器/传感器引脚 → 提示"此值是本台硬件的映射标定结果,需开环实测辨识,换硬件须重辨识,请确认本台接线/安装/旋向"。
- **闭环极性主动核查——基于用户确定的映射验证**:用户确认各传感器/执行器映射方向后,agent **主动核查**在该映射下闭环是否为负反馈,不只是转述。核查方法:沿反馈链逐级核对符号,确认 **Kp × 反馈斜率 < 0** 成立、各级无隐藏取反、Kp 全正号;若用户确定的映射与代码现状矛盾(如用户说"正命令=前进"但 `*_OUTPUT_DIR=-1"),**指出矛盾并请用户复核接线/宏取号**。
- 整定闭环控制器(尤其 yaw/航向)→ 提示"闭环轴向对齐开环辨识不覆盖,请手动转动确认传感器正向=被控量正向"。
- 改动跨越数量级的增益/步长 → 提示"此增益作用于 X 量纲,与 Y 环不可混用;步长须与范围同量级,请确认"。

**极性/传感器方向独立列入硬件映射(`configHeader` §极性段 / manifest `headers.polaritySource`)**,不混进普通参数,便于审查时一眼定位。

## 五、IO 外设归属(设计期放置,交叉引用 hf-embedded-safety)

设计/登记涉及**单实例 IO 外设**(SPI 屏、共享总线、共享 ADC、调试 UART 等)的模块时:

- **主动提醒该外设的多核归属权,促用户做分核任务规划**:哪个核/上下文拥有、其它核如何让出、冲突如何仲裁。
- 登记到 manifest `targets[].ownedPeripherals[]`:`device` / `owner` / `io:true` / `gate:"whitelist-#if"` / `planNote`(`io:true` 即多核归属敏感)。
- **门控机制本身**(白名单 `#if`、失控锁定、volatile)归 `hf-embedded-safety`——本 skill 只负责**设计期放置 + 归属提醒 + manifest 登记**,不重复门控细节。

## PASS/FAIL 对抗审查清单

逐项 PASS/FAIL + 依据,任一 FAIL 先修再交付:

- [ ] **引脚集中**:本次涉及的引脚全在 `pinMap.h` 唯一定义,无散落各 `.c` 的硬编码。
- [ ] **pinMap 零依赖**:`pinMap.h` 不 `#include` 驱动头,纯 `#define`。
- [ ] **三层分离**:算法/控制层不直接写引脚/寄存器;引脚走 pinMap、参数走 configHeader。
- [ ] **极性唯一归属**:所有方向翻转只在 §极性段三组 `*_DIR`;**PID Kp 全正号**,无负号藏极性。
- [ ] **不变量成立**:每实例 正命令→正方向 + 正转速反馈 + 正电流反馈,三组 DIR 自洽。
- [ ] **DIR 经辨识**:`*_DIR` 的 ±1 标注"开环实测辨识/换硬件重辨识",不是照抄的固定常量。
- [ ] **闭环轴向**:有闭环环(yaw/航向)时,传感器正向=被控量正向已**手动确认**并注释,不只靠开环辨识。
- [ ] **量纲不混**:新增/改增益明确作用量纲,未跨环照搬(速度域 vs 电流域)。
- [ ] **步长同量级**:可调参数步长与默认/范围/钳位同量级。
- [ ] **configHeader 分节**:参数有 §分节索引 + 量纲约定注释,§极性段独立。
- [ ] **PINOUT 同步**:改了 pinMap 已(或提示)同步跨核引脚冲突矩阵。

## 验证(agent 能做的 + 必须交给用户的)

- agent 能做:静态判定头组织/极性归属/数量级量纲、把散落引脚收进 pinMap、把藏 Kp 的极性搬回 §极性段(属改行为,须按 `hf-refactor` 对抗审查)、登记 manifest、生成提醒文案。
- **必须交用户**(物理事实,agent 无法验证):每个 `*_DIR` 的 ±1 实测辨识、闭环轴向转车确认、增益整定的板端阶跃响应。agent 必须给出"上板须辨识/确认 X"的明确交接,**不得替用户假定极性**。

## 反面教训(具体案例)

- **极性藏进 Kp 负号**(头号):把执行器极性翻转写成 PID Kp 的负号,Kp 兼增益+极性。某次误改符号→负反馈变正反馈→执行器跑飞。根治:极性全归 §极性段三组 DIR,Kp 恒正。
- **编码器 INDEX↔引脚接线辨识**:源工程 `core2` BL/BR 两轮重新接线后,极性随之翻转;若照抄旧 `ENCODER_*_DIR` 则反馈符号全错,60A 强磁电机闭环即失控。教训:`*_DIR` 是**本台硬件**的辨识结果,重接线必重辨识,改前核实物理接线。
- **菜单步长与范围数量级失配**:给 Kp=0.002 量级的电流环参数配了与速度环(Kp~23)同一档的步长,运行时按键一下直接越过整个有效区间。教训:步长随参数量级走,4 处历史失配。
- **闭环轴向漏确认**:只做开环单轮辨识就上 yaw 闭环,未确认 IMU `yawRate` 正向=车体左转,导致航向环方向反。教训:开环辨识不覆盖整车轴向,须手动转车确认。
