# hf-refactor — 行为保持重构 / Behavior-Preserving Refactor

agent 跑不了目标工具链(IAR/Keil 等),"行为不变"不能靠编译+运行验证。本 skill 把重构约束成**有限的机械等价变换**,并按 `../hecateflow/references/orchestration-contract.md` 先自主求证、主动派只读子代理对抗审查 + 复审链代替编译,逐处证明位级等价。适用"重构/去重/精简/复用已有库且不改变行为";**不适用**改算法、调参、加功能(那些本就要改行为,走常规流程)。

## 触发 / 不触发

- 触发:重构、去重、合并相同逻辑、精简、充分利用已有库/抽象。
- **不触发**:改 bug、调参、改算法、加功能。若用户要改行为,不要套本 skill。

## 第一性原则:只允许这 6 类变换

1. **提取重复为助手/库调用**:多处相同逻辑 → 一个函数,调用点展开后与原版字符级等价。
2. **复用已有抽象**:本仓已有库(clamp/wrap/PID/低通/施密特/数学工具)或现成外部库能用就用,不重写。重点查"为复用而设计却被手写绕过"的抽象;不足 650 行但跨 target 高频复用的能力,也可在零行为变化前提下抽成公共库或公共头。
3. **合并生命周期**:重复的 Init/Reset 字段清零合并,字段集合逐字段相等(不多清/不漏清)。
4. **删确证死代码**:全工程零引用且非库 API 的 static 函数/变量、孤立废注释。
5. **删被取代的函数**:被新实现完全替代的旧函数。
6. **行为保持分文件**:自写文件超过 650 行的结构性拆分,或超过 1000 行默认拆分/封装管理的场景,可把硬件底层、硬件顶层、软件实现拆成多个 `.c/.h`;调用顺序、数据流、static 可见性、初始化顺序和外部接口保持不变。vendor/generated/table 或用户确认的特殊长文件例外不强拆。

**不做**(过度抽象,保持现状):单次使用的逻辑块、参数类型各异的并列函数、一行惯用法(饱和计数/临界区)、需编译期常量检查的规整初始化。判据:抽象后**调用点更短且语义更清晰**才做;只是把重复挪个地方还要跳转理解 → 不做。

## 等价性红线

- **符号/极性不能被通用助手吃掉**:取反 `clamp(-x,...)` 不能换成无取反的助手——**头号红线**。
- **前馈/钳幅表达式**:运算、顺序、中间量必须等价。
- **生命周期字段集合**:逐字段相等。
- **赋值顺序 / volatile 读语义 / 整型 cast 链 / 溢出行为**:不变。
- **轴/通道映射、控制流、分频、调用顺序**:不串、不变。

## 辨识"有意保留",严禁误删

下列**不是死代码**:通用库对外 API(零调用但是储备能力)、待接入链路(标注"后续接入"、未进构建)、调试开关 gated 分支、对称储备符号(对称于在用符号)。删除前逐项排除。**跨 target 同名文件(motor/IMU/PID)语义不同,禁止跨 target 合并。**

## 执行流程

1. 锁定 target + scope,确认是"零行为变化"(否则退出本 skill)。
2. 搜现有库/抽象(复用优先于新抽象)。
3. 把计划的每处改动归类到 6 类变换之一;归不进的不做。若是分文件,先列出原文件行数、拆分后文件职责、外部接口是否不变、构建登记点。
4. 逐处实施。
5. 实施后**派只读子代理对抗审查**:喂下方清单,用 `git show HEAD:<path>` 与工作区逐函数比对,要求逐项 PASS/FAIL + 依据。
6. 派复审视角检查子代理证据充分性、矛盾和过度推断;主 agent 亲自 grep/读码复核涉及删除/改行为的关键点。
7. 任一 FAIL、CRITICAL/HIGH 或影响等价性的 MEDIUM 先修,再派只读复审;反复迭代到所有复审子代理 PASS,或剩余项属于 A3 用户确认/物理验证/平台限制。
8. 全 PASS 才请求 Git 确认;未经用户确认不 stage/commit/push。

## 对抗审查清单(子代理逐项 PASS/FAIL)

- [ ] 提取的助手/库调用展开后与原版**字符级等价**(运算/顺序/中间量)。
- [ ] **符号/极性未被通用助手吃掉**(取反不能用无取反助手替换)——头号红线。
- [ ] 前馈/钳幅表达式等价。
- [ ] 逐处变量名核对(N 处替换,每处赋值回的是原操作的同一变量)。
- [ ] 生命周期函数字段集合逐字段相等(不多清/不漏清)。
- [ ] 控制流/分频/调用顺序未变。
- [ ] 轴/通道/外设映射未串。
- [ ] volatile 读语义、整型 cast 链、溢出行为不变。
- [ ] 删除的函数/变量确证全工程零引用且非库 API；设计注释保留。
- [ ] 子代理报告经复审链核证据/矛盾/过度推断,主 agent 已亲验关键点。
- [ ] 分文件重构保持行为:硬件底层/硬件顶层/软件实现拆分后调用顺序、初始化顺序、static 状态归属、外部 API、构建登记均与原行为等价;>1000 行特殊长文件不拆时已有用户确认例外。
- [ ] 高频复用小模块(<650 行)已评估是否应抽为公共库/公共头;若抽取,展开后行为等价且构建登记同步。
- [ ] 复审迭代闭环完成:FAIL/CRITICAL/HIGH/影响等价性的 MEDIUM 已修复并重新复审,当前 change set PASS 或剩余 A3 项已列明。

## 验证

- 纯等价改动:告知用户"目标工具链编译通过即证明等价、运行应逐拍一致、无需重新整定参数"。
- 接入新库/改安全逻辑:给出具体上板验证步骤。
- agent 无法编译这一限制必须显式交接,不假装已验证。

## 反面教训

- "顺手改个常量":破坏零行为变化——重构提交里夹带行为改动,等价性失守。
- 过度抽象:把并列清晰的 6 个共享内存读写函数压成一个宏参数化,可读性反降。
- 删"没人调用"的库 API:它是储备能力,删了破坏库完整性。
- 轻信子代理"零引用"结论:某次 Explore 报某宏"零用"与"9 用"自相矛盾,亲验实为 2 处真调用,轻信则误删致编译断裂。
- 复审子代理指出等价性缺口后只解释不重审:没有重新 PASS 的重构不能当作完成。
- 大文件拆分顺手改接口:本来只是把千行文件拆层,却顺手改了单位/返回值/初始化顺序,不再是行为保持重构。
